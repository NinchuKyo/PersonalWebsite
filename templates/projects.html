This Site
Code
The reasoning as to why I undertook this project to create a personal site was to showcase the various projects that had worked on before, and to show what skills I can bring to a potential person or organization.

Technologies Used

This site is hosted on AWS (Amazon Web Services), using an EC2 instance. Web page requests / responses are handled by the Apache module mod_wsgi as the interface for Flask (a Python micro web framework) responsible for processing requests received and returning the appropriate web pages. The front-end framework used is Materialize, which allows for a Bootstrap-like ease-of-use, but with a different flavour.

Design Decisions

Originally, this site was hosted on Heroku, but I then chose to switch over to AWS due to constraints present on Heroku's free tier, which made it for web applications must sleep for 6 hours / day. With Heroku however, the features provided made it easy to deploy web applications as it could pull from GitHub automatically whenever a commit was made.

Also, I did not have to worry about setting up Apache, or configuring security settings. However, what I wanted was full uptime without having to worry if my site was down, and for payment to be as a per-use basis, instead of a flat fee on Heroku. The other bonus was gaining the knowledge of how to deploy simple web applications using AWS.

Flask was chosen as a web framework due to its simplicity as it comes with the bare essentials for making a web application (not needing a lot of boilerplate code to get the job done), and the ability to get a small site like mine with minimal features up and running in no time.

I decided to use Materialize for the front-end framework as I wanted something that looked stylish, but different as to what Bootstrap had offered, which was responsive web design. What really impressed me about Materialize was the use of cards, which I felt was perfect for displaying little tidbits of information, but giving a bit of insight on what the card was about through the card image. If a person wanted to know more about the card topic, then links could be provided below the card, just like what the cards on the home page accomplished by taking you to this page (if you used them to get to this page).

Future Project Plans

The great thing about persuing this project is the potential for improvements. In the future, I'm thinking of maybe adding a blog section about video games and technologies so I can discuss my thoughts and opinions about them. By doing so, I hope to improve my writing ability for professional purposes and write about topics that I feel passionate about.

_Note was actually a university course project that I cooperated with five other classmates to make. We named it _Note, as the blank could stand for whatever you needed it to be used for, be it for students needing to store and access their synced notes from a desktop or mobile iOS device, or a busy professional on the go for mobile only.

At the time, my classmates and I were introduced to an app called Evernote, and were amazed at how it could be used seamlessly across a multitude of devices. So we thought, "Hey, wouldn't it be cool if we could make something like that?", and thus became the inception of _Note. Due to the size of this project, each member had a specific role.

In the project, my role involved anything that was iOS related. This meant that my main goal were getting the iOS design and development up and running, utilizing the testing frameworks for iOS, and making sure that it would work with our server design. I felt that this was an interesting challenge as before this, I had never worked with iOS app development nor using Objective-C.

_Note
Code
Technologies / Methodologies Used

Considering that this project involved more than one person, we needed a means of managing ourselves and how software development should be done. For the software development methodology, Agile was used. We used Trello as our Kanban board for user stories and feature planning. The project lasted for three iterations, with each iteration being about two to three weeks long.

The web framework / back-end used for _Note was Flask and hosting was done through AWS using an EC2 instance (the same as what was used for this website), but instead of the standard responses returning static html pages, _Note uses a REST-style architecture for its responses, where they are made in JSON in order to support both desktop browsers and mobile applications. MVC was the driving software architecture on how the application was structured. For version control, Git was used, with GitHub used to host our Git repository for everyone on our team to access.

The front-end desktop browser client design was done using a bit of custom CSS, Bootstrap, and AngularJS. For the mobile application, which was an iOS application, Objective-C, with the Cocoa framework was used. Storyboards were used to manage the multiple view controllers, like having a slide-out menu for the list of notes, and a view to load the note's content when a list item is tapped.

Note storage was handled by using the user's Dropbox account that they allowed the application to store notes via OAuth, which was handled by the Dropbox API.

For testing the desktop browser client, QUnit was used for unit testing, and Selenium was used as the acceptance testing framework. As for the iOS client, we used QUnit for iOS, and for acceptance testing, we used Frank.

Design Decisions

In this case, Flask was chosen for this project, as before this, we had never used Python or Flask for back-end web development and decided as a group that we wanted to try it out. In terms of only making an iOS application and not an Android application as well, was that it was not required by the course.

When we were wondering what technologies to use for the desktop browser front-end, one of the members of our team had experience working with AngularJS and Bootstrap. Since none of the other members had used this before, we wanted to learn more about it. I feel that one of the best ways to learn something new is to leverage someone else's knowledge, as they went through the process of learning it, so they would know the in's and out's of the technology and how to do things.

Since none of us had worked with mobile development technologies, we just went with what was recommended to us in the class. It was certainly a different experience compared to what I had been used to programming with C#, especially with the use of NS prefixes in the language, and how to manipulate objects and variables. Use of the storyboard felt similar to Visual Studio's WinForms, so I felt it was pretty easy to get used to.

In terms using the Dropbox API to store a user's notes through Dropbox, we felt that based on our application being related to notes accessibility and modifications across different platforms, we felt that this the perfect technology to use. Originally, we wanted some form of syncing across platforms. However, writing code that handles syncronization is extremely difficult. Instead, we thought about how Dropbox was able to sync files across platforms, so maybe we can somehow leverage it to store and sync notes, and sure enough they had an API which we could use.

For the testing framework that we chose to use, we just went with the ones we felt was easier to use, and the ones that were recommended to us. In class, we were recommended a couple of options for which testing frameworks to use. For acceptance testing of the browser, we had the option of using Jasmine or Selenium, and as noted above, we went with Selenium.

Lessons Learned

Overall, I felt that from doing this project, I learned a lot on how programming in a larger group can be difficult sometimes, and what ends up happening a sometimes were communication issues, where one member doesn't communicate to others what they are doing, resulting in duplicated work or end goals aren't communicated clearly enough, resulting in wasted work. Having went though this experience taught me greatly on the value of effective communication and the impact it could have when working with a team.

Sometimes there were moments where there were disagreements, but a great mentor that I had the pleasure of working with imparted me with some great advice, which was, "At the end of the day, you just brush it off and don't take anything said against you personally". A part of being a software developer is listening to different opinions and being able to acknowledge how others think and to "walk a mile in their shoes". In the end, our goal is to make someone else's life easier through our work.

SharePoint Reporting Tool
Code
This project is one of the sole reasons for the quote that you see on this page. While I was working for this one company, then subsequently for another company after, a lot of the work involved SharePoint development.

Some of the projects for these two companies involved gathering and modifying information relative to an item within a list or a library. Instead of constantly having to go on the specified SharePoint site to gather this information, or to change the information, I wanted to develop a tool that could make this work easier for the other projects that I was working on. This was especially important for the SharePoint Migration Project where the items from SharePoint 2007 differed from SharePoint 2010.

Technologies Used

The SharePoint Reporting Tool is a desktop WinForms application built using Visual Studio 2012, under the .NET 4.5 Framework, with the SharePoint 2010 SDK.

Design Decisions

I felt that for anything that is Microsoft-related, the best tools for working with their own products are usually also from Microsoft as I feel that their product integration is usually pretty good. Since SharePoint is a Microsoft product, I figured they probably had tools for developers to use in order to access and modify SharePoint list items. So sure enough, they had the SharePoint SDK, which was perfect for my needs.

The idea behind this tool was to make it simple and efficient for me to use, but with enough features such as logging to make it easy to debug unexpected errors. Tabs were used to divide each area with a specific task. For example, the first tab was used solely for selecting which lists or libaries to work with, and of which SharePoint site they were working with.

The second tab focused soley on the information contained within a list or library, and provided logical operations for finding items within a list. Other features included displaying only a selection of metadata values of items (sometimes items had way too many metadata values, some of which weren't useful), customizing your own query for list items, and exporting the results to Excel (which was very useful, as it worked with the other programs that I had). These features were developed to provide more information for me to use in conjunction with other programs.

The third tab was made primarily for logging, where it was primarily made for visibility behind what errors could occur, especially when dealing with custom data fields in SharePoint lists. This feature was especially useful during the development of this program.

The reasoning as to why WebForms was used, was that it was the quickest possible solution in terms of solving my immediate needs at the time. Considering that at the time I only knew C# WebForms, and wanted something to help me with my other projects, I felt that it was the perfect choice.

Lessons Learned

From doing this project, I learned that by using C# I am able to automate a lot of tasks that normally would seem tedious if done by hand. This project led me to learn more about the other libraries that I could use, in order to automate work.

An example of where I was able to apply this new knowledge was during one of my work terms. By using C#'s Microsoft.Office.Interop library in order to generate an automated Word report, which was populated with embedded Excel documents, I was able to reduce the amount of time spent writing up the report by hand.

This was one of the very cool things that I found that I could do with C#, and because of this, I am always looking for a way to make my work easier if I could write a program to do it for me.
